What no university will teach…but is VITAL to your career!

If you’re either wrapping up your college program or 12-week bootcamp (or just did), then you’re probably all fired up and stoked to GET OUT there! Yeah?

Ready to carve out your own piece of this thing we call Software Development.

Would it SURPRISE you to know that one of THE MOST IMPORTANT things you could learn to enhance your career was totally skipped over? One of those VITAL skills any pro coder needs is simply nowhere to be found in any 4-year collegiate program nor any 12-week bootcamp?

What is this magical skill we all need but isn’t being covered?

The answer: Continuous Integration

Continuous Integration (CI) is when you let a server build your software after every commit.

  • If it succeeds, it shows GREEN
  • If it fails it shows RED.
  • Some CI servers will email you. Others may send you an IM. But the point is that you let this CI server run every single change anyone makes.

Why is this so valuable?

And more importantly, why didn’t anyone TELL YOU about it during your studies?

The reason CI is so valuable is that we need feedback, CONSTANT feedback on whether or not our changes have broken anything. Even if we made a seemingly tiny change, having a CI server run our tiny change and detect breakage will help by STOPPING us and suggesting we back up and see what went wrong.

But that is not all.

CI servers don’t just build our software. They can do much more. If our code is consumed downstream, then publishing a snapshot to a repository will make it much easier for others to eat up our changes.

And…it makes it easier to BREAK them.

You heard me. Breaking downstream consumers of our module is GOOD because it exposes our lingering PROBLEMS sooner rather than later.

The Spring team is BIG on backward compatibility. If Spring Data JPA were to drop a publicly-visible method that Spring Boot takes advantage of, the sooner we can push that into their team, the better. They’ll either accept this change, or they’ll ring us up, asking “What happened?!?”

Okay, the Spring Boot team won’t REALLY react that way. But they would certainly inquire as to WHY we dropped a publicly visible API in the middle of a point release.

CI is good at catching stuff sooner rather than later. And there is a hint of possibility that your formal education in software development MAY have mentioned, the sooner bugs are caught, the less they cost to fix.

Which brings us back to WHY you’ll find NONE of this in your training!

Because they don’t CARE.

Or to be more precise…they care about a lot of stuff including: algorithms, language constructs, variables & functions, logical tactics, and development tools.

Maybe they DO care about other things like version control, CI, and operations, but not ENOUGH to bump any of these other topics. Relative to programming concepts and coding practices, no instructor views code hygiene as something they can fit into an already busy schedule.

And so it becomes something YOU must embrace and learn about. Something YOU must decide will help you stop FIGHTING MIDNIGHT FIRES. Something YOU must choose to pick up and learn to do.

Because in my 25-year career as a software engineering veteran whose first assignment was testing OTHER people’s code (and hence built an automated CI tool before CI had entered today’s lexicon), nothing has helped me MORE than knowing SOONER rather than LATER that a particular code change broke something.

And the sooner I knew it, the sooner I could fix it.
