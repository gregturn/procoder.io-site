What is the good, the bad, and the ugly of CI?

What is the good, the bad, and the ugly of CI?

CI has lots of nooks and crannies. What does it take to pull things together that best serves you?

What is the good, the bad, and the ugly of CI?

If there’s one thing I do, it’s jumping in feet first when I find something I like. A keen example is: Docker.

I don’t mean everything in Docker. Quite the contrary. I find the usage of Docker for development actually gets in my way. No, I’m talking about using Docker as a core piece of building any continuous integration or CI solution.

I’m a big fan of building every single commit pushed to github. And by “building”, what I mean is, “test against a matrix of compatible profiles, publish the artifacts to a repository, and generate documentation”.

The Problem

The Spring portfolio has a strong reputation of backward compatibility. To ensure this on the projects I manage (Spring Web Services, Spring Session for MongoDB, and Spring HATEOAS), I need to test each version against multiple versions of the Spring Framework, Spring Security, and Spring Data.

This was a chore nonetheless when we used the old CI system (Bamboo by Atlassian). You went into a web page, configured all kinds of stuff, then hoped for the best. It sort of worked. But every time you wanted to make a change, you had to revisit the web site and learn it all over again.

  • Want to upgrade Maven? Contact the sysadmin.
  • Want to install a different version of Java? Contact the sysadmin.
  • Want to use a particular OS? Contact the sysadmin.

See the pattern? Trying to do something edgy, like testing against multiple versions of the JDK was out of the question. No way do any of us have the time for that.

Several Solutions

This is the reason I wholeheartedly embraced Travis CI years ago when it emerged. Visit their website and enable your project. That’s it! All you must do is add a single file with build details to your code base, and you’re set.

With a huge growth in Travis, other similar CI systems emerged. One that really caught my fancy was CircleCI. The biggest difference was your ability to specify the Docker container your builds would run inside. That meant pulling down images from Docker hub, not just what the CI company offered.

That came in handy when I spotted minor versions of Java breaking Spring Web Services due to tiny wrinkles inside the JDK’s SOAP libraries.

That’s when I realized Docker was the ticket to CI nirvana. Docker combined with the Maven wrapper would absolutely obliterate those CI configuration issues.

The Spring Data release train has grown to 15 officially supported modules. Team leaders Oliver Drotbohm and Mark Paluch had built a tool to automate releasing this beast, but a small gap had grown huge — the Bamboo jobs set up long ago to monitor each module couldn’t keep up.

There was no way to spot, for example, that someone’s patch in Spring Data Commons would break Spring Data REST due to a change rippling through Spring Data JPA. Discovering these sorts of issues on the day of a release was no way to operate.

A Challenge

So Oliver put out the call for someone to attack our CI system. He didn’t care what the solution was, just that it had to not be rocket science so we all could handle it.

I raised my hand. This sort of server-side, infrastructure honing is one of my “things.” And so for several months, I hammered away at building a pipeline.

Did I mention pipelines? Historically, CI systems used to be these mammoth single-job runners. But certain shops (cough Netflix) raised the bar by defining pipelines where smaller tasks would get chained together. And so other CI systems began defining pipelines. CircleCI introduced one that I used for certain things. But it was clunky and hard to reason what was happening.

That’s when I discovered Concourse. Oh my, talk about an elegant solution. Everything is boiled down to a job with a set of tasks. And every task is a Docker-managed script with inputs and outputs. You chain together a series of tasks where one task’s outputs are the next task’s inputs. VERY easy to reason about.

I rapidly added parallel jobs that would compile different profiles against different versions of Java. That, my friend, was the ticket toward creating confidence.

But there were still a few wrinkles. The Spring Data team must publish branched artifacts without stomping on released ones. Anytime an official branch is pushed to github, we needed a pipeline to run. And Concourse flat out could NOT handle that. You configured it to monitor one and only one branch.

Revisiting an Old Enemy

So after trudging through thick and thin, I had to put Concourse on the shelf and turn to my old nemesis: Jenkins.

Jenkins is arguably the oldest CI system you’ll ever find. And it has a history of big, clunky configuration screens with pages and pages of options. This, to me, was worse than Bamboo!

But Jenkins in the past couple years had apparently adopted a pipeline approach. And had adopted a Groovy-based DSL. And (drum roll) had introduced 1st class support for Docker.

Jenkins also has about 3–4 different dialects to configure a job. I sought out the purist, simplest DSL approach I could find. I had to scrap certain features. But the idea was to make a given version of a given Spring Data module as simple to read as possible. So my teammates would embrace it fully.

And so I have finally come to accept that the true solution for CI is Jenkins (at least for me). Why? For these key reasons:

  • Every stage run inside a Docker container (except actually building containers).
  • Pipeline captured as code in the project.
  • Ability to run certain stages in parallel with conditional settings.
  • Easy to read and comprehend, hence easy to alter as needed.

If you ever bump into me at a conference, be sure to say “hey!”
